02 — Project Idea
Why This
System Exists
The Problem

Millions of tons of food are wasted daily while many communities lack access to adequate nutrition. Traditional food redistribution channels are manual, slow, and uncoordinated — resulting in spoilage and missed connections between donors and recipients.

The Solution

A layered, intelligent software system that connects food donors with beneficiaries in real time — using AI-powered matching, a conversational chatbot, and data analysis to optimize every redistribution decision.

The Goal

Reduce food waste. Maximize redistribution efficiency. Empower communities through a production-grade, modular software architecture.

SFR // Layer 1: UI/UX 05 / 12
LAYER 1 — UI / UX DESIGN
Design Before Code
Purpose of this layer
The UI/UX layer defines the entire user experience before a single line of code is written. It answers: What will users see? How will they navigate? What visual language guides the product?
wireframes/
Low-fidelity sketches of each screen. Defines layout, content placement, and navigation — without colors or styling. Focuses on structure only.
mockups/
High-fidelity visual designs with real colors, fonts, and spacing. Represents the final look of the application before development.
user-flows/
Diagrams showing how a user moves through the system — from login to donation submission to confirmation. Maps the complete user journey.
design-system/
A shared library of colors, typography rules, spacing scales, and reusable UI components. Ensures visual consistency across the entire app.
figma-links.md
References to live Figma files where the team collaborates on design in real-time. Single source of design truth.
Design-to-Dev Pipeline

UI/UX defines the contract between design and engineering. Frontend developers implement exactly what UI/UX specifies — reducing ambiguity and rework.

Key Outputs
Screen Layouts User Journey Component Library Visual Style
Flow to Next Layer

UI/UX designs are handed off to the Frontend layer, which implements them using HTML, CSS, and JavaScript.

SFR // Layer 2: Frontend 06 / 12
LAYER 2 — FRONTEND CLIENT
The User's Window
Purpose of this layer
The Frontend is everything the user sees and interacts with directly. It renders the UI, handles user events, sends HTTP requests to the backend, and displays results — all in real time.
index.html
The structural skeleton of the application. Defines all DOM elements, semantic layout, and includes external scripts and stylesheets. Entry point of the frontend.
index.css
All visual styling — layout, colors, typography, responsive breakpoints. Implements the design system defined in the UI/UX layer. Controls how everything looks.
index.js
Client-side logic. Handles button clicks, form submissions, API calls to the backend (fetch/axios), dynamic DOM updates, and state management.
Frontend Responsibilities
Render HTML views for all app screens
Capture and validate user input from forms
Call backend REST API endpoints via HTTP
Display API responses dynamically in the DOM

Key Principle

Frontend is stateless — it holds no permanent data. All real data lives in the backend database. Frontend simply displays and requests.

SFR // Layer 3: Backend 07 / 12
LAYER 3 — BACKEND CORE
The Brain of the System
Purpose of this layer
The Backend is the central processing unit of the entire system. It receives requests from the frontend, executes business logic, coordinates with AI modules, reads/writes from the database, and returns structured responses.
app.py
Flask application entry point. Initializes the server, registers all route blueprints, configures middleware, connects to the database, and starts the application.
routes/
Each file defines a group of API endpoints (e.g., /donations, /users, /matches). Routes accept HTTP requests, validate input, call services, and return JSON responses.
services/
Contains the core business logic. Services are called by routes and perform operations like: match a donation, notify a beneficiary, process a request. Independent and testable.
models/
ORM data models that map to database tables. Define the schema for: User, Donation, Request, Match, Notification. Used by services to read/write persistent data.

SFR // Layer 4: AI Engine 08 / 12
LAYER 4 — AI INTELLIGENCE ENGINE
Smart Decision Making
Purpose of this layer
The AI layer adds intelligence to the system. Instead of hard-coded rules, it uses algorithms to make smart decisions: Who should receive a food donation? How should the chatbot respond to a user query? This is what makes the system "smart".
matching.py
The core matching algorithm. Takes a food donation (type, quantity, location, expiry) and a set of beneficiary requests, then calculates the best match using scoring criteria: proximity, urgency, food type compatibility, capacity. Returns optimal pairings.
chatbot.py
A conversational assistant that helps users interact with the system through natural language. Handles: "How do I donate food?", "What donations are near me?", "Check my request status". Uses NLP to parse intent and return helpful responses.
How Backend Calls AI
backend/services/match_service.py
from ai.matching import find_best_match
result = find_best_match(donation, requests)
Matching Algorithm Flow
Input
Donation + all active requests
PROCESS

RANK
Output
Best match returned to backend
Chatbot Capabilities
  • Answer FAQs about the platform
  • Guide users through donation flow
  • Report on request status
  • Collect user feedback
Key Principle

AI layer is called by the backend. It never communicates with the frontend directly. This keeps AI logic clean and independently testable.

SFR // Layer 5: Data Analysis 09 / 12
LAYER 5 — DATA ANALYSIS
From Data to Insights
Purpose of this layer
After the system runs, we need to understand: Is it working? Are donations being matched efficiently? Where are the bottlenecks? The Data Analysis layer answers these questions and feeds insights back to optimize the AI and backend logic.
analysis.ipynb
Jupyter Notebook for exploratory data analysis. Contains Python code to load, clean, visualize, and analyze data. Produces charts and statistics about system usage, match rates, donation patterns, and regional demand.
datasets/
Raw and processed data exported from the system database. Includes donation records, beneficiary requests, match outcomes, user activity logs — anonymized for analysis.
reports/
Auto-generated or manually compiled reports summarizing key metrics. Shared with stakeholders to track KPIs: total redistributed food (kg), match success rate (%), average response time.
insights.md
A markdown document of written conclusions from analysis. E.g.: "Matching rate drops on weekends" or "Bakery donations expire before matching". Directly informs AI improvements.

Analysis results feed back into the AI layer to retrain and tune the matching algorithm — making the system smarter over time.

SFR // Layer 6: Documentation 10 / 12
LAYER 6 — DOCUMENTATION
The System Knowledge Base
Purpose of this layer
A system without documentation is a system only its creator understands. The Docs layer captures all knowledge: how the API works, how the system is designed, what was planned and why. It enables onboarding, maintenance, and collaboration.
api.md
Complete API reference documentation. Lists every endpoint: URL, HTTP method, request parameters, request body format, response format, status codes, and example responses. The contract between Frontend and Backend.
diagrams/
Visual system representations: ER diagrams (database schema), sequence diagrams (how components communicate), architecture diagrams (system overview), and flowcharts (matching algorithm). Makes complex systems understandable at a glance.
Plan proposal.pdf
The original project proposal document. Contains: project objectives, scope definition, timeline and milestones, team roles, technical approach, and expected deliverables. The foundational planning artifact.
Root Config Files
.env.example
Template for environment variables: DB connection strings, API keys, secret keys. Developers copy this to .env and fill in real values.
requirements.txt
Python dependency list. Run pip install -r requirements.txt to install all packages: Flask, SQLAlchemy, scikit-learn, pandas, transformers.
